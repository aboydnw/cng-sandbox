# Frontend Gotchas

Implementation details and non-obvious behaviors in the frontend. Read this when working on map rendering, tile sources, client-side COG rendering, dialogs, analytics, or anything that touches `src/lib/layers/`, `src/components/PixelInspector.tsx`, `src/hooks/`, or the vector/raster map components.

## Tile URLs and loaders

- **Vector tile sources need absolute URLs**: `buildVectorLayer` (`src/lib/layers/vectorLayer.ts`) prefixes `window.location.origin` onto relative tile URLs before handing them to deck.gl's `MVTLayer`. External PMTiles go through `/api/proxy` to avoid CORS.
- **deck.gl handles relative URLs fine**: Raster tiles via `createCOGLayer`/`useTitiler` work with relative paths since deck.gl uses `fetch()` internally.
- **SSE named events**: The ingestion API sends `event: status` SSE events. The frontend must use `addEventListener("status", ...)`, not `onmessage` (which only handles unnamed events).
- **COG tiler proxy preserves `/cog` prefix**: Unlike `/raster` and `/vector` (which strip their prefix before forwarding), the `/cog` proxy passes the path through unchanged because titiler's COG routes are already mounted under `/cog/`.
- **Vendored maptool utilities**: `src/lib/maptool/` contains `createCOGLayer`, `createPMTilesProtocol`, `useColorScale`, `MapLegend`, and `listColormaps` — vendored from `@maptool/core` so the sandbox has no external dependency on the library.
- **Categorical rasters use a JSON colormap, not `colormap_name`**: When a dataset has `isCategorical: true`, tile URLs are built with `colormap=<encoded-JSON>` (a `{value: [r,g,b]}` map) and `resampling=nearest` instead of the usual `colormap_name=` parameter. Mixing them will produce incorrect or broken tiles.
- **Vector tiles must parse on the main thread**: `buildVectorLayer` (`src/lib/layers/vectorLayer.ts`) needs two things together: `loaders: [MVTLoader]` AND `loadOptions: { worker: false }`. MVTLayer's default loader is `MVTWorkerLoader`, which has no main-thread `parseSync` — so disabling the worker alone produces `"mvt loader - no parser found and worker is disabled"`. Registering the full `MVTLoader` gives it a main-thread parser. The worker must still be disabled because loaders.gl's default MVT worker fetches its script from `unpkg.com`, which the Caddy CSP `script-src` blocks in prod. The inner `load(..., MVTLoader)` call used for PMTiles also passes `worker: false`. Keep both unless we start self-hosting the loaders.gl worker bundle and widen the CSP accordingly.

## Client-side COG rendering

- **Client-side COG rendering is the default when eligible**: Eligibility is centralized in `evaluateClientRenderEligibility` (`src/lib/layers/clientRenderEligibility.ts`). A COG item is eligible if it has a `cogUrl`, valid `bounds` within ±85.05° latitude, is not temporal, has a single band (multi-band COGs fall back to server-side rendering), and has a file size within the browser cap (paletted: 2 GB; continuous: 500 MB). `useMapControls` initializes `renderMode` from `item.renderMode` (the server-persisted value on the dataset/connection row): if the stored mode is `"client"` and the item is eligible, it uses client; if the stored mode is `"client"` but the item is ineligible, it falls back to `"server"`; if no mode is stored, eligible items default to `"client"` and ineligible items default to `"server"`. When the user manually switches modes, the choice is persisted server-side via `PATCH /api/datasets/{id}/render-mode` or `PATCH /api/connections/{id}/render-mode` — not to localStorage. `useRasterOverrides` continues to persist colormap and rescale settings to localStorage, but no longer stores `renderMode`. The render-mode toggle is hidden on shared (read-only) map views. Layer construction is centralized in `resolveRasterLayers` (`src/lib/layers/resolveRasterLayers.ts`), which both `useLayerBuilder` (for `MapPage`) and `buildLayersForChapter` (for the story reader/editor) delegate to — stories go through the same eligibility path as maps. `resolveRasterLayers` exposes a `forceServer` escape hatch (tested in its own suite) for callers that want to bypass client rendering unconditionally; no production caller currently uses it. `resolveRasterLayers` re-checks eligibility, then (unless `forceServer` short-circuits to server tiles) calls `classifyCogRenderPath` (`src/lib/layers/cogDtype.ts`) on the source's `dtype` + `isCategorical` flag. `uint8`/`int8` or any integer dtype flagged categorical → `buildCogLayerPaletted`. When categories are supplied, the paletted builder runs a LUT-driven shader pipeline (`CreateTexture` + `Colormap` modules) using a 256-entry RGBA LUT from `buildCategoricalLut` (`src/lib/layers/categoricalLut.ts`), attaches the raw integer tile values to the tile's data so the pixel inspector can sample them, and respects user-edited category colors; without categories it falls back to the library default pipeline (honours the file's color table). Everything else → `buildCogLayerContinuous` (float32 → uint8 normalization + viridis colorize shader, attaches raw float tile values to the tile's data for the pixel inspector). For connections without a stored rescale, the continuous builder falls back to `[0, 1]` (see `parseRescaleString` in `src/lib/connections/rescale.ts`). The cap logic reads `converted_file_size` for datasets and `file_size` for connections. Changing the classifier rules only requires updating `evaluateClientRenderEligibility` and/or `resolveRasterLayers` — both call sites pick up the new behavior automatically.
- **deck.gl-raster upgrades require wrapper audits**: The custom client-side COG builders in `src/lib/layers/cogLayer.ts` pass raw tile arrays into deck.gl-raster GPU modules so the pixel inspector can keep sampling original values. When upgrading `@developmentseed/deck.gl-raster` or `@developmentseed/deck.gl-geotiff`, read the upstream release notes/issues and re-check these wrappers for obsolete compatibility code. In particular, deck.gl-raster 0.5 fixed deck.gl 9.3 texture upload behavior, so local row-padding/byte-alignment workarounds around `device.createTexture({ data, format: "r8unorm", width, height })` should not be reintroduced.
- **Render mode is surfaced in every map view**: `RenderModeIndicator` (`src/components/RenderModeIndicator.tsx`) is an info button in the top-right of the map that reveals the active render mode and why, so users can see whether a layer is rendering client-side or falling back to server tiles. `MapPage` renders it next to `SnapButton` for raster items, pulling eligibility via `evaluateClientRenderEligibility(item)` and the size from `dataset.converted_file_size` / `connection.file_size`. Stories use the same component: `buildLayersForChapter` returns `{layers, renderMetadata}` where `renderMetadata` carries `{renderMode, reason, sizeBytes}`, and `StoryRenderer` / `StoryEditorPage` render the indicator from that metadata. The indicator carries `data-snapshot-overlay` so it's included in map snapshots.
- **Pixel inspector has two tooltip modes**: `usePixelInspector` (`src/components/PixelInspector.tsx`) reads raw values off the hovered tile's own `content.data` (populated by both client-render builders on `getTileData` return) and returns a discriminated `hoverInfo` union. Sampling directly from the hovered tile — rather than a shared `(x, y)`-keyed cache — means zoom-level collisions can't return stale values from a different tile. When the active layer is categorical _and_ the user is in client render mode, `MapPage` passes the effective categories to the hook, which matches the looked-up value against the category list and emits a `{kind: "categorical"}` tooltip (color swatch + label via `CategoricalPixelTooltip`). Otherwise it emits a `{kind: "numeric"}` tooltip (formatted value + coord via `PixelInspectorTooltip`). Both tooltips are rendered in `MapPage` and gated on `renderMode === "client"`.
- **Pixel inspector is projection-aware for non-Mercator COGs**: `lookupValue` in `PixelInspector.tsx` projects the cursor's lng/lat through `data.projectFrom4326` (lng/lat → source CRS) and then through the per-tile `content.inverseTransform` (source CRS → pixel `[col, row]`) before indexing into `raw`. Both helpers are populated by the COG layer builders: `buildCogLayerPaletted` / `buildCogLayerContinuous` (`src/lib/layers/cogLayer.ts`) stash a `proj4("EPSG:4326", sourceProjection)` forward function on each tile's data once `COGLayer.onGeoTIFFLoad` resolves the source CRS, and deck.gl-geotiff already attaches `inverseTransform` at `tile.content`. The earlier implementation linearly interpolated lng/lat across the tile's lng/lat AABB — exact for identity projections and OK for Web Mercator at small spans, but visibly wrong for projections with curved parallels (NLCD Albers in CONUS shifts by ~15 px / ~450 m at typical zooms; polar stereographic and others would be worse). A lng/lat-linear fallback path remains for tiles without a projector — server-tile mocks and the brief window before `onGeoTIFFLoad` fires.
- **COG URL resolution handles dataset and connection sources**: `resolveCogUrl` in `src/lib/layers/cogLayer.ts` passes absolute `http(s)://` URLs through unchanged (external COG connections) and prefixes `window.location.origin` onto relative paths (dataset COGs served through the Vite proxy). Both client-side builders route their COG URL through this helper before handing it to the deck.gl COG layer.

## Snapshots, GeoParquet, DuckDB

- **Map snapshot composites WebGL canvases + DOM overlays**: `useMapSnapshot` (`src/hooks/useMapSnapshot.ts`) draws the MapLibre basemap canvas and the deck.gl canvas onto an off-screen `<canvas>` via the shared `compositeMapCanvases` helper (`src/lib/story/archival/compositeMapCanvases.ts`), then uses `html-to-image` to rasterize any elements marked `data-snapshot-overlay` (e.g. the map legend) and composites them at their correct positions. The output is downloaded as a PNG. Elements that should appear in snapshots must carry the `data-snapshot-overlay` attribute.
- **GeoParquet connections support two render paths**: `render_path: "client"` loads the file into DuckDB-WASM via `useGeoParquetRender`, returns an Arrow `Table`, and renders via deck.gl (500k feature cap). `render_path: "server"` triggers a background conversion (tippecanoe → PMTiles → R2); the frontend `useConnectionConversion` hook subscribes via EventSource (SSE) to `GET /api/connections/{id}/stream` and shows a conversion overlay on `MapPage` until the job finishes and a `tile_url` is available. When `render_path` is omitted, the server infers it: files over 50 MB default to `"server"`, smaller files default to `"client"`. The frontend also runs `pickRenderPath` (size + 500k feature threshold) to pre-select the path before submitting the connection.
- **DuckDB-WASM initializes on first use**: `useDuckDB` initializes lazily — the WASM bundle and worker are loaded the first time `useGeoParquetRender` calls `initialize()`. `useDuckDB` deduplicates concurrent/subsequent `initialize()` calls so only one instance is ever created.

## UI library

- **Chakra v3 dialogs need Portal + DialogPositioner**: `Dialog.Content` must be wrapped in `<Portal><Dialog.Positioner>...</Dialog.Positioner></Portal>` to render as a fixed-position centered modal. Without them, content renders inline at its JSX-tree location (e.g. squished inside a flex header). See `ShareDialog.tsx`, `PublishDialog.tsx`, `UploadModal.tsx`, `GeoParquetPreviewModal.tsx` for the canonical pattern.
- **Chakra v3 toaster**: A minimal toaster scaffold lives in `frontend/src/lib/toaster.ts` (exports a singleton `toaster = createToaster(...)`) and `frontend/src/components/ui/toaster.tsx` (renders `<Toaster toaster={toaster} />`). The `<Toaster>` component must be mounted once in `App.tsx`. Use `toaster.create({ title, description, type })` anywhere in the app to show a toast — do not call `createToaster` again elsewhere.

## EPSG resolver

- **COG layers use a bundled EPSG resolver**: All COG layer builders (`buildCogLayerPaletted`, `buildCogLayerContinuous`) pass `epsgResolver: cngEpsgResolver` to `deck.gl-geotiff`'s `COGLayer`. The resolver (`src/lib/layers/epsg/resolver.ts`) checks an inline registry first (built from `subset.csv` + a UTM zone generator in `src/lib/layers/epsg/inlineRegistry.ts`), then falls back to the `@developmentseed/proj` default resolver which fetches `https://epsg.io/{code}.json` over the network. The inline set covers ~30 common codes plus all 120 UTM zones, so most real-world COGs never touch the network.
- **Network fallback is CSP-coupled in prod**: The fallback resolver fetches from `https://epsg.io`. If a deployment CSP restricts `connect-src` to exclude that origin, COGs in uncommon CRSes will fail to render (the resolver promise rejects and tiles stay blank). Common CRSes — Web Mercator, NAD83/Conus Albers, all UTM zones, WGS84 geographic, common European and polar projections — are in the inline registry and are unaffected. The current `Caddyfile` CSP has a permissive `connect-src`; keep this in mind when tightening it.
- **Vitest requires `@developmentseed/proj` to be inlined**: The proj package ships extensionless relative imports (e.g. `from "./parse-wkt"`) that vitest's Node ESM resolver rejects. `vite.config.ts` lists it under `test.server.deps.inline` so Vite transforms the package instead of passing it to Node. Do not remove this entry without verifying the full test suite still passes.

## Portable story config and iframe embeds

- **Portable rendering skips the backend entirely**: When `StoryReaderPage` receives a `?config=` search param, it calls `loadPortableConfig` (`src/lib/story/loadPortableConfig.ts`) to fetch or decode the `cng-rc.json`, then passes the result to `cngRcToStory` (`src/lib/story/cngRcAdapter.ts`). The adapter synthesizes `Connection` objects (keyed `portable-<layerKey>`) from the layer metadata and assembles a full `Story` in memory — no API calls to the ingestion backend. The resulting `connectionMap` is passed directly to `StoryRenderer`.
- **Connection synthesis rules**: `cngRcToStory` iterates `config.layers`, picks `source_url ?? cng_url` as the tile URL, and calls `synthesizeConnection`. Layer type maps to `connection_type` (`raster-cog` → `"cog"`, `pmtiles` → `"pmtiles"`, `xyz` → `"xyz_raster"`, `vector-geoparquet` → `"geoparquet"`) and `tile_type` (`raster-cog`/`xyz` → `"raster"`, `vector-geoparquet` → `"vector"`, `pmtiles` → `null`). All synthesized connections are `render_path: "client"` and `conversion_status: "ready"`.
- **COG layers in portable mode are a known gap**: The viewer subdomain has no `/cog/` proxy, so portable `raster-cog` layers fall back to the server-tile path (`/cog/tiles/…`), which returns nothing from the viewer subdomain. They will only render once either (a) `cng-rc.json` includes layer bounds so client-side rendering can engage, or (b) the exporter emits absolute COG tile URLs. v1 reliably supports vector-geoparquet, pmtiles, and xyz.
- **Inline config via `base64url:` prefix**: `loadPortableConfig` accepts either an `https://` URL (fetched at runtime) or a `base64url:<encoded>` string (decoded in-browser). Inline payloads are capped at 100 KB. The base64url variant uses `-` and `_` instead of `+` and `/` and omits padding; `loadPortableConfig` re-pads before calling `atob`.
- **Embed route and viewer subdomain**: The embed snippet (generated by `EmbedSnippet`/`PublishDialog`) points at `<VITE_VIEWER_ORIGIN>/story/:id/embed?config=<url>`. The `/story/:id/embed` path renders `StoryReaderPage` with `embed={true}` (hides the title bar). The viewer subdomain (`VIEWER_SITE_ADDRESS` in `.env`) serves the same static SPA bundle with no basic auth and `frame-ancestors *` in the CSP so the iframe can load anywhere. `VITE_VIEWER_ORIGIN` must be set at build time to the full viewer subdomain URL (e.g. `https://viewer.<domain>`); if omitted, `PublishDialog` falls back to `window.location.origin` (the auth-gated main domain) and end users will see a login prompt. Full setup is in `docs/production-deployment.md`.

## Static bundle export and viewer build target

- **The viewer bundle is a separate Vite build**: `yarn build:viewer` (`vite build --config vite.viewer.config.ts`) compiles `viewer-entry.tsx` + `viewer.html` into `frontend/public/viewer/` as a single self-contained HTML file (via `vite-plugin-singlefile`). All JS, CSS, and inlineable assets are inlined; geotiff decoder worker chunks that cannot be inlined are emitted as content-hashed sidecar files beside `viewer.html`. The full build runs `yarn build:viewer` first, then `tsc && vite build` for the main app — the viewer output must exist before the main build so the `public/viewer/` directory is included in the static bundle.
- **`public/viewer/` is gitignored**: `frontend/public/viewer/` is listed in `.gitignore`. The directory is regenerated at build time; never commit its contents. `frontend/public/.vite/` (Vite's internal dep-optimization cache) is also gitignored.
- **`manifest.json` is the runtime file list**: `emitFileManifest` (a custom Rollup plugin in `vite.viewer.config.ts`) writes `public/viewer/manifest.json` listing every file name emitted into `public/viewer/` (sorted). `buildAndDownloadBundle` (`src/lib/story/buildStaticBundle.ts`) reads `GET /viewer/manifest.json` at runtime to know which files to fetch and zip. If you add an output format or otherwise change what `vite.viewer.config.ts` emits, the manifest plugin will pick it up automatically — no manual changes needed.
- **Any change to `viewer-entry.tsx`, `viewer.html`, or anything they import transitively will produce new content-hashed filenames** for sidecar chunks in `public/viewer/`. The manifest is regenerated on each build, so the bundle builder always gets the current filenames. However, a stale build artifact (old `public/viewer/` left from a previous build) will serve the wrong files until `yarn build:viewer` is rerun. In Docker the `RUN yarn build` step always regenerates from scratch.
- **Static bundle download flow**: `buildAndDownloadBundle` fetches every file listed in `manifest.json` plus `GET /api/stories/{storyId}/export/config` (the `cng-rc.json`), then zips them with JSZip. `viewer.html` is renamed to `index.html` inside the zip. The result is downloaded as `<slugified-title>.zip`. The bundle is self-contained: unzip and open `index.html` from any static file server; `viewer-entry.tsx` loads `cng-rc.json` via `fetch("./cng-rc.json")` relative to the HTML file. There is no backend dependency at runtime.
- **`vite.shared.config.ts` holds config common to both builds**: worker format, esbuild target, `optimizeDeps.exclude`, and the resolve deduplication list. Both `vite.config.ts` and `vite.viewer.config.ts` merge from `sharedConfig`. Keep shared settings there to avoid drift.

## Archival HTML export

- **Archival HTML is a standalone export format**: `buildArchivalHtml` (`src/lib/story/archival/buildArchivalHtml.ts`) generates a single self-contained HTML file from `cng-rc.json`. All external dependencies — map snapshots, images, videos, CSS — are inlined as base64 data URLs or embedded in `<style>` tags, making the result safe to archive without any external file references. Chapter assets (maps via `captureChapterMap`, videos/images via `fetchAndInlineAsBase64`) are fetched and base64-encoded at build time.
- **Dublin Core metadata is included**: `dublinCoreMetaTags` embeds `DC.title`, `DC.creator`, `DC.issued`, `DC.identifier` and other standard metadata in the document's `<head>`, enabling digital preservation systems and archive tools to index the story correctly.
- **Map snapshots are captured dynamically**: `captureChapterMap` (`src/lib/story/archival/captureMap.tsx`) mounts a non-interactive `UnifiedMap` into an off-screen 1200×675 host (positioned at `left:-10000px`), waits for the MapLibre `idle` event plus a 250ms quiet period after the last deck.gl `onAfterRender` with all layers reporting `isLoaded`, then composites the basemap and deck.gl canvases via `compositeMapCanvases` and encodes the result as a base64 PNG data URL. A 30s timeout aborts capture if the map never settles. `UnifiedMap` is now mountable without the app's providers (the archival path passes `enableSnapshot: true` and `hideBasemapPicker: true`). Chart chapters are marked pending (chart snapshots are not yet captured in v1). Scrollytelling chapters render a single snapshot (not the full scroll experience). Zarr chapters are rejected with an explicit error (zarr chapters need the React `useStoryZarrNode` hook, which the offscreen capture tree can't host yet).
- **Snapshot capture uses the live `datasetMap`/`connectionMap`, not `cngRcToStory`**: `captureChapterMap` accepts `datasetMap` and `connectionMap` so each chapter renders through the same code path as the in-page reader — ingested datasets route through `/raster/collections/{id}/...` (server-side titiler-pgstac) instead of being fed back through `cngRcToStory`, which always sets `render_path="client"` with the raw `source_url`. The previous synthesized-story path 500'd on example datasets whose `source_url` is a bucket directory listing.
- **The `@deck.gl/react` ref must be dereferenced lazily**: `captureChapterMap` stores the entire imperative handle (`refs.deckHandle`) and reads `refs.deckHandle.deck.canvas` only at canvas-read time. `@deck.gl/react@9` calls `useImperativeHandle` with an empty deps array, so the parent's ref callback fires exactly once (during commit, before the `useEffect` that creates the underlying `Deck` instance runs). The handle exposes `deck` via a live getter, so the value updates over time, but if the parent eagerly unwraps `handle.deck` inside the ref callback it freezes the value at `undefined` and `captureChapterMap` later throws "Capture canvases not available after map ready". The in-app `useMapSnapshot` avoids this naturally by using a `RefObject` and reading `deckRef.current.deck.canvas` at snap time.
- **The readiness gate must be re-checked from both inputs**: `captureChapterMap`'s gate (`mapIdle && allLoaded`) has two asynchronous inputs — deck.gl's `onAfterRender` and maplibre's `'idle'` event — and either may fire last. The shared `checkReady()` helper is invoked from both callbacks; evaluating the gate only inside `onAfterRender` deadlocks when `'idle'` arrives after deck.gl's final draw (no further render to re-observe `mapIdle`), and the capture sits at the 30s timeout with both inputs satisfied. The ref callback also short-circuits `mapIdle = true` when `map.loaded()` already returns true at subscribe time, because `once('idle')` never fires for a basemap style that was cached and settled before subscription.
- **Citations are collected and inlined**: `buildCitationBlock` collects all dataset citations from the story's layers, dedups them, and renders a structured citations section with DOI links where available. Each dataset that contributes to the story has its provenance visible in the final HTML.
- **Download and progress tracking**: `downloadArchivalHtml` (`src/lib/story/archival/downloadArchival.ts`) fetches `GET /api/stories/{storyId}/export/config` alongside the live `Story`, `Dataset[]`, and `Connection[]` from `/api/stories/{id}`, `/api/datasets`, and `/api/connections`, builds `datasetMap`/`connectionMap`, then calls `buildArchivalHtml` and triggers a browser download. It accepts an optional `onProgress` callback (used by `ArchivalProgress` in `PublishDialog`) to track conversion progress across chapters. Errors surface to the user via the Chakra toaster from `useArchivalDownload`, not just the console.
- **Image and video chapters use wrapped `extra.image` / `extra.video` objects, not flat fields**: `cng-rc.json` stores image chapter media under `extra.image = { url, thumbnail_url, alt_text }` and video chapter media under `extra.video = { provider, video_id, original_url }`. The wrapping happens in `_extract_chapter_extra` in `ingestion/src/services/story_export.py`, which keys the live `ImageChapter.image` / `VideoChapter.video` payloads under their chapter type. The archival renderer in `buildArchivalHtml.ts` must read these wrapped shapes — earlier versions referenced flat `extra.image_url` / `extra.video_url` / `extra.thumbnail_url` keys that don't exist in real configs, which silently rendered empty `<img>` tags. Video thumbnails for `provider: "youtube"` are derived from `video_id` via `https://img.youtube.com/vi/<id>/maxresdefault.jpg`; non-YouTube providers (e.g. Vimeo) have no auto-thumbnail and fall back to a placeholder. When either asset fails to inline, the renderer emits a visible `<p><em>(Image/Video unavailable)</em></p>` placeholder rather than an empty element, and `tryInlineAsset` logs the underlying error via `console.warn` so failures aren't silent.

## Interactive HTML export (server-side bundle)

- **The interactive bundle is built server-side and downloaded as a zip**: `downloadInteractiveExport` (`src/lib/story/interactive/downloadInteractive.ts`) fetches `GET /api/stories/{id}/export/config`, `GET /api/stories/{id}`, `/api/datasets`, and `/api/connections`, then captures one PNG per **scrollytelling** chapter via `captureChapterMap` (the same off-screen capture helper as the archival flow). It POSTs the PNGs as multipart `scrolly_pngs` to `POST /api/stories/{id}/export/interactive` and saves the returned `.zip` as `<slugified-title>.zip`. Static map chapters are *not* captured client-side — the server's archive-runtime bundle renders them live from the manifest's tile/Arrow sources.
- **`useInteractiveDownload` mirrors `useArchivalDownload`**: `src/lib/story/useInteractiveDownload.ts` exposes `handleInteractive`, `handleCancelInteractive`, and a `progress` object compatible with `ArchivalProgress`. `ExportDialog` and `PublishDialog` both pass `onInteractive` into `ExportSection` and render a second `ArchivalProgress` modal driven by the interactive hook. Errors surface to the user via the Chakra toaster titled "Interactive export failed", not just the console.
- **Manifest carries the full styling envelope, not just `rescale`**: Layer manifests emitted by `ingestion/src/services/interactive_export/builder.py` now include `opacity`, `colormap_reversed`, `rescale_min`, `rescale_max`, `band`, and `timestep` (the legacy `rescale: [min, max]` tuple stays as a backwards-compat fallback when only Plan 1 manifest fields are present). The archive-runtime workspace (`frontend/archive-runtime/`) honors them: `MapChapter.ts` calls `rescaleFromLayer` (prefers `rescale_min/max`, falls back to the tuple), `BitmapLayer`/GeoArrow layers honor `opacity`, and the raster shader honors `colormap_reversed` via `sampleColormap`/`applyColormapToTile`. Editing this contract requires keeping `builder.py` and `frontend/archive-runtime/src/types.ts` in sync.
- **Chart chapters are resolved server-side before the builder runs**: The endpoint pre-resolves each chart chapter via `src/services/interactive_export/chart_data.py` (`resolve()` dispatches by `source.kind`): `csv_asset` reads R2 bytes (mirrors `story_assets.py`), `dataset_timeseries` calls `cached_timeseries` from `routes/dataset_charts.py`, `dataset_histogram` calls `titiler_statistics`. Because of this, the three helpers in `routes/dataset_charts.py` were renamed from underscore-prefixed (`_load_dataset`, `_cached_timeseries`, `_titiler_statistics`) to public (`load_dataset`, `cached_timeseries`, `titiler_statistics`). The builder receives a `chart_data_by_chapter` dict and rejects chart chapters with no resolved payload.
- **Image and video chapter media are inlined into `assets/<chapter>-image.<ext>`**: `src/services/interactive_export/asset_inline.fetch_into` pulls bytes through `validate_url_safe` (same SSRF rules as `/api/connections`), so private/loopback IPs and unsafe schemes are blocked. YouTube thumbnails are derived from `video_id`; other providers fall back to whatever URL is in the manifest. When inlining succeeds the manifest emits `image_src: "assets/..."`; on failure it falls back to the raw `image_url` so the archive-runtime can still attempt a remote fetch at view time.
- **Chapter bboxes are intentionally coarse**: `_bbox_for_chapter` in `builder.py` derives an EPSG:4326 bbox from the chapter's saved `map_state.center`/`zoom` covering ~4 tiles of context. This over-pads on purpose to avoid the complexity of exact tile-bound math; a tighter heuristic is flagged `TODO(plan 4)`. If raster pyramids look like they're tiling too much area, this is why.

## Chart chapters (CSV, timeseries, histogram)

- **CSV charts have editor and reader modes**: `ChartChapterRenderer` receives an optional `onRangeChange` callback. When present (editor mode via `ChapterPreview`), the chart renders all rows, enables interactive dataZoom controls (slider + inside), and debounces zoom events to persist the `x_min`/`x_max` range to the chapter's `viz` config. When absent (reader mode via `StoryRenderer`), the chart filters rows by the saved `x_min`/`x_max` before rendering and pre-applies that range as `startValue`/`endValue` to the dataZoom controls, so users see the curated x-axis window by default but can still zoom out to the full dataset. Timeseries and histogram chapters ignore range persistence.
- **X-range persistence is axis-type-aware**: `filterRowsByRange` in `ChartChapterRenderer.tsx` handles numeric (min/max comparison), date-string (ISO parse), and category (label index slice) x-axes. When the user zooms the chart, `handleDataZoom` reads `startValue`/`endValue` for numeric/date axes and falls back to label slicing for category axes — this ensures the stored range survives datasource changes and can round-trip correctly.
- **dataZoom slider is hidden in read-only mode**: When `onRangeChange` is absent (shared/read-only stories), dataZoom includes only `{ type: "inside" }` (scroll-to-zoom), omitting the slider UI. The editor (`ChapterPreview`) passes `onRangeChange` so it always shows both controls.

## Zarr connections

- **Consolidated metadata is required for auto-enumeration; manual path fallback exists**: `probeZarr` (`src/lib/zarr/probeZarr.ts`) calls `zarr.withMaybeConsolidatedMetadata` and checks if the returned store exposes a `contents()` method. If it doesn't (no `.zmetadata` for v2 or no `consolidated_metadata` block in `zarr.json` for v3), `probeZarr` throws with `ZARR_NOT_CONSOLIDATED`. `ConnectionModal` catches that sentinel and reveals a "Manual variable path" input — the user types a path (e.g. `precipitation` or `group/var`) and clicks "Try this path", which calls `probeZarrSingleArray(url, path)`. `probeZarrSingleArray` (also exported from `probeZarr.ts`) opens the array directly without needing a consolidated index, classifies it, and returns a single-variable `ZarrProbeResult` that feeds `ZarrConnectionFields` exactly as a full probe would. Both `probeZarr` and `probeZarrSingleArray` are exported from the same module.
- **CORS fallback is per-origin and cached**: `createZarrStore` (`src/lib/zarr/zarrFetch.ts`) wraps `zarr.FetchStore` with a custom `fetch`. On the first request to an origin it tries a direct browser fetch; if a `TypeError` is thrown (the browser's CORS error surface), it falls back to `/api/zarr-proxy?url=<encoded>` and records `"proxy"` for that origin in module-level state. All subsequent chunk requests for the same origin skip the probe and go straight to the proxy. Use `createZarrStore` everywhere zarrita needs a store — never construct `zarr.FetchStore` directly.
- **MVP dimensionality constraint**: `classifyVariable` in `probeZarr.ts` marks a variable compatible if it has exactly 2 dimensions (plain 2D spatial), exactly 3 dimensions where the third is recognized as a time dim (`time`, `t`, `valid_time`) or as a fixed extra dim (`band`, `level`, `channel`, `depth`, `z`, `pressure`, `height`, with size ≤256), or exactly 4 dimensions where one is a recognized time dim and another is a recognized extra dim. Variables with 1 dimension, 5+ dimensions, or unrecognized non-spatial dims are marked incompatible with a descriptive reason. `ZarrConnectionFields` disables incompatible options in the variable `<select>` and renders a band/level/channel picker when the selected variable has an extra dim.
- **Stats are parsed from CF attrs**: `extractStats` (`src/lib/zarr/zarrStats.ts`) reads rescale min/max from `valid_min`+`valid_max` (CF convention) or `actual_range` (2-tuple), in that priority order. If neither attr is present the fields are left blank and the user must enter them manually. `scale_factor` and `add_offset` are intentionally not applied — the rescale range is expressed in physical units; the renderer applies those attrs itself.
- **Time dimension decoding and decimation**: When a variable has a recognized time dim, `probeZarr` opens the coordinate array and decodes it to ISO strings using the `units` attr (format `"<unit> since <epoch>"`). Coords longer than `MAX_TIME_STEPS_DECODED` (5000) are decimated to ~5000 evenly-spaced samples (always including the first and last index) so we don't bloat the persisted connection config or the time slider. Each decoded value is stored as `{ datetime, index }` in `config.timesteps` — `index` is the actual zarr time-coord index (NOT the slot position), which matters because decimation makes the two diverge. If decoding fails, `timesteps` is `null` and `ZarrConnectionFields` shows a "raw indices" notice instead of the step count.
- **Connection payload for zarr**: `ConnectionModal` POSTs `connection_type: "zarr"` with `config: { variable, timeDim, timesteps, extraDim, extraIndex, rescaleMin, rescaleMax }` (where each timestep is `{ datetime, index }`; `extraDim` is the recognized non-spatial dim name like `band` or `level` and `extraIndex` is the user-selected coordinate index along it) plus an optional top-level `geozarr_attrs` object when the user fills out the Advanced — GeoZarr metadata override section. `extraDim`/`extraIndex` are present only for variables that carry an extra dim. The submit button stays disabled until `ZarrConnectionFields` calls `onConfigChange` with a non-null config (a compatible variable is selected and valid min < max are entered).
- **GeoZarr metadata override**: `ZarrGeoZarrAttrsFields` (`src/components/ZarrGeoZarrAttrsFields.tsx`) is a collapsible "Advanced" section in `ConnectionModal` for entering the four single-resolution GeoZarr keys (`spatial:dimensions`, `spatial:transform`, `spatial:shape`, `proj:code`) when a store is missing inline GeoZarr metadata. The same component appears as an inline editor on `ConnectionInfoCard` for existing zarr connections, which calls `connectionsApi.setGeoZarrAttrs` → `PATCH /api/connections/{id}/geozarr-attrs` to update the persisted override. The override is passed verbatim to `buildZarrLayer` as `geozarrAttrs` and forwarded to deck.gl-zarr's `ZarrLayer` via the `metadata` prop, where `@developmentseed/geozarr.parseGeoZarrMetadata` consumes it. Backend validation (`src/services/geozarr_attrs.py`) mirrors the frontend's parser so bad payloads return 422 instead of crashing the renderer.
- **Render path for zarr connections**: `useZarrNode(item)` (`src/hooks/useZarrNode.ts`) opens the store via `createZarrStore` + `withMaybeConsolidatedMetadata` and resolves the root group. When the connection's `config.variable` resolves to a zarr Array (the common single-resolution case, e.g. IMERG `precipitation`), `useZarrNode` pre-opens that array and returns it as the node; `useLayerBuilder` then passes the Array directly to `buildZarrLayer` and omits the `variable` prop. This sidesteps `ZarrLayer._parseZarr`'s `zarr.open(root.resolve(variable), { kind: "group" })` call, which throws `NotFoundError: Not found: v2 group` when the variable path is an array rather than a multiscale group. For multiscale stores (`config.variable` resolves to a Group), `useZarrNode` returns the root group and the layer resolves the variable group itself, as designed. The opened node (Array or Group) is threaded into `useLayerBuilder` as `zarrNode`, which calls `buildZarrLayer` (`src/lib/layers/zarrLayer.ts`). Coloring uses the `Colormap` GPU module from `@developmentseed/deck.gl-raster/gpu-modules`, fed by a 256-entry RGBA LUT built by `buildContinuousLut` (`src/lib/layers/continuousLut.ts`) from the `lib/maptool/colormaps.ts` palette set. As a result, the existing colormap picker (driven by `useMapControls.colormapName`) drives zarr rendering identically to COG, and the existing temporal slider/animate UI works because `connectionToMapItem` populates `MapItem.isTemporal` + `MapItem.timesteps` from the saved `config.timesteps`. When fetching a frame, `useLayerBuilder` reads `item.timesteps[slot].index` to get the real zarr coord index — the slider slot is just a position into the (possibly decimated) list, so the slot integer cannot be used directly as the time-dim selection. For variables with an extra dim (e.g. 4D `(time, level, y, x)` or 3D `(band, y, x)`), `useLayerBuilder` also slices the extra dim using `item.zarrExtraDim`/`item.zarrExtraIndex` (populated from `config.extraDim`/`config.extraIndex` by `connectionToMapItem`), so the user's band/level/channel pick is applied alongside the time selection.
- **`Colormap` requires a 2D-array LUT texture**: the deck.gl-raster `Colormap` shader module declares `uniform sampler2DArray colormapTexture` and indexes layers via `colormapIndex`. The texture spec (`dimension: "2d-array"`, `depth: 1`, `width: 256`, `height: 1`, `format: "rgba8unorm"`) is centralized in `createColormapLutTexture` (`src/lib/layers/colormapLutTexture.ts`); both `buildZarrLayer` (linear filter, continuous) and `buildCogLayerPaletted` (nearest filter, categorical) call it. A plain 2D texture will silently fail to bind. If you ever swap to a different colormap helper, keep the array dimensionality and update the shared helper rather than reintroducing the spec inline.
- **No overview pyramid means low-zoom = many chunk fetches**: Unlike COG (which has built-in overviews) or PMTiles (which carries pre-rendered tiles), zarr stores expose only their native chunks. Zarrita pulls each chunk that intersects the viewport at the requested zoom level, so a global zoom-out over a high-resolution grid can trigger hundreds of small `Range` requests. Zarrita has a `maxRequests` limit to bound concurrency but no multi-scale fallback. Multi-scale zarr (overview pyramid) is out of scope for the URL-connection track — see Track 2 of the design spec for the eventual fix. For MVP, prefer modestly sized stores or warn users that low-zoom rendering is slow.
- **Story rendering uses a separate zarr hook**: The interactive map viewer uses `useZarrNode` (driven by `useLayerBuilder`). Story rendering — both `MapChapter` and `StoryRenderer` — uses `useStoryZarrNode` (`src/hooks/useStoryZarrNode.ts`) instead. `useStoryZarrNode` takes a raw `Connection | null`, opens the store with `createZarrStore` + `withMaybeConsolidatedMetadata`, pre-resolves the variable array when `config.variable` points at an array (same Array-vs-Group disambiguation as `useZarrNode`), and returns a `ZarrNode`. The caller builds a `zarrNodeMap` keyed by connection ID and passes it as the fourth argument to `buildLayersForChapter`, which then calls `buildZarrLayer` directly. This path avoids the `MapItem`/`connectionToMapItem` abstraction used by the interactive viewer, so timestep and extra-dim selection are read directly from `layer_config` (e.g. `lc.timestep`, `lc.rescale_min`) and the connection's persisted `config` fields.

## Vector layer RGB color mode

- **`color_mode: "rgb"` on `LayerConfig` enables PCA-style vector coloring**: When a chapter's `layer_config` sets `color_mode: "rgb"` and `color_property: "<property>"`, `buildLayersForChapter` injects a `getFillColor` accessor into the vector layer that reads the named property from each feature. The property is expected to be a 3-element array of floats in `[0, 1]` (e.g. PCA-derived RGB embeddings). Each component is clamped to `[0, 1]`, scaled to `[0, 255]`, and alpha is fixed at 200. Features missing the property or with a non-array value fall back to `[128, 128, 128, 200]` (medium grey). This override is applied in all three vector paths: `geoparquet` connections, `pmtiles`/`xyz_vector` connections, and inline vector datasets. Only `color_mode: "rgb"` is implemented; other values are ignored. Both `LayerConfig` (frontend) and `LayerConfigPayload` (backend model) carry the two new optional fields.

## Analytics

- **Plausible analytics is loaded from two places**: the `<script async>` tag in `frontend/index.html` pulls the tracker from `https://plausible.io` and `initPlausible()` in `src/lib/plausible.ts` (called from `main.tsx`) installs a command-queue stub on `window.plausible` and invokes `init()`. The stub lets us call `window.plausible(...)` before the remote script finishes loading; events are flushed once it arrives. Keep the CSP `script-src` entry for `https://plausible.io` in `Caddyfile` in sync with the script tag — removing one without the other will silently break analytics in prod.
