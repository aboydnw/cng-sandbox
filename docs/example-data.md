# Example Datasets and Stories

Read this when touching `src/services/example_datasets.py`, `src/services/example_stories.py`, `src/services/example_connections.py`, `src/services/example_workspace.py`, the `is_example` / `is_example_copy` flags, or the source.coop seeding logic.

## Per-workspace example copies (current model)

The `is_example=True` rows (datasets, stories, connections) are a **master set**. They are no longer surfaced directly in a workspace's `GET /api/datasets|stories|connections` list responses. Instead, each workspace gets its own **editable clones** of the master set, flagged `is_example_copy=True`.

- `src/services/example_workspace.py` ā€” `seed_workspace_examples` clean-slate clones every master into a workspace (datasets ā†’ connections ā†’ stories), rewriting story chapter `dataset_id`/`connection_id` references to the cloned ids; `remove_workspace_examples` deletes those clones. Cloning is metadata-only: clones reuse the master's `tile_url`/`stac_collection_id`/`source_url`, so no pgSTAC/R2 work happens and deleting a clone is a shallow DB-row delete (never tears down shared storage). The individual `DELETE /api/datasets/{id}` route and the cleanup sweep both special-case `is_example_copy` for this reason.
- `POST/DELETE/GET /api/workspaces/{id}/examples` ā€” seed (also "add back"), remove, and report state. State (`seeded`/`removed`) lives in the `workspace_example_state` table so auto-seed fires once for a new workspace and a workspace that removed examples is not re-seeded.
- The frontend calls `POST ā€¦/examples` when it mints a brand-new workspace id (landing-page buttons + home-redirect mint), and the Data page exposes a Remove/Add example-data toggle.
- The public `GET /api/stories/examples` endpoint still reads the master rows so the landing carousel works without a workspace.

The master `is_example` rows themselves are still seeded exactly as described below.

## Master seeding (source.coop products)

On startup, the ingestion service runs a background task (`src/services/example_datasets.py`) that registers the curated source.coop products as shared "example" datasets. These rows carry `is_example=True`, are owned by no workspace, cannot be deleted or modified via the API, and serve as the clone source (and the public-carousel source) so a fresh deploy never has an empty library. The task is idempotent across restarts (it skips products whose `listing_url` is already present on an example row) and registers fast products before slow ones so the gallery populates quickly. PMTiles-kind products are registered via `src/services/pmtiles_register.py`, which probes the first 127 bytes of the remote file (`src/services/pmtiles_header.py`) to extract bounds, min/max zoom, and verify the tile type is MVT; mosaic-kind products are enumerated and registered via `register_remote_collection`.

Example dataset primary keys are **deterministic**: each row's `id` is `uuid5(EXAMPLE_DATASET_NAMESPACE, source_url)`, derived via `example_dataset_id(source_url)`. This guarantees that a database wipe + re-seed produces the same dataset IDs, so any story chapters (in example stories or forks) that reference them keep resolving. **Never change `EXAMPLE_DATASET_NAMESPACE` or the way the ID is derived** ā€” doing so would orphan every example dataset row and every chapter referencing one. A startup migration (`migrate_example_dataset_ids`) renames legacy example rows that still carry their historical random uuid4 ID to the deterministic one; `tile_url` and `stac_collection_id` are left alone because they were baked at original registration time and continue to function as opaque tokens against the existing pgSTAC collection.

Stories also support the `is_example` flag (see the `stories` table). Example stories are **not** surfaced directly in a workspace's `GET /api/stories` response ā€” each workspace instead owns editable clones (`is_example_copy=True`) produced by `seed_workspace_examples` (see the per-workspace section above). The master `is_example=True` rows themselves cannot be modified or deleted via the API. They are still exposed via the unauthenticated `GET /api/stories/examples` endpoint so the public landing page can render the example carousel without provisioning a workspace. A separate startup task (`src/services/example_stories.py`) polls on a fixed cadence and calls `seed_example_stories`, which inserts any story whose required example datasets are already registered (matched by `source_url` in dataset metadata). Running independently of the dataset-registration retry loop means stories whose datasets are ready can seed without waiting for slower products (e.g. GHRSST temporal enumeration). Seeding is idempotent ā€” stories whose required datasets aren't yet registered are retried on the next tick. Existing example rows (matched by title) are **refreshed in place** (description, `dataset_id`, `chapters_json`) rather than skipped, so a re-seed always rewrites chapter `dataset_id`s from the current seed catalog.

Before the seeder runs, `relink_dead_chapter_dataset_ids` walks every example story and every explicit fork of an example story (rows with `forked_from_id` set) and rewrites only those chapter `layer_config.dataset_id`s that no longer resolve in the `datasets` table. The seed catalog is the source of truth: an example story is matched to its `StorySeed` by title, and a fork is matched via the parent's title (`forked_from_id`), falling back to the fork's own title. **User-authored stories are skipped entirely** even if their title happens to coincide with a seed ā€” rewriting user chapters from the seed would be destructive. Live user-edited references that still resolve are preserved, and rows whose chapter count has diverged from the seed are skipped to avoid position-based mis-replacement.

Connections also support the `is_example` flag (added on the `connections` table alongside the existing flag on `datasets` and `stories`). Master example connections are owned by no workspace and cannot be modified or deleted via the API; like datasets and stories they are **not** surfaced directly in `GET /api/connections` but are cloned into each workspace as `is_example_copy=True` rows. A separate startup task (`src/services/example_connections.py`) seeds entries from the hardcoded `EXAMPLE_CONNECTIONS` list ā€” idempotent on `(url, connection_type)`. The list currently ships five entries: IMERG Final Precipitation (a global half-hourly precipitation zarr), Fields of The World (a global 4D field-prediction zarr that exercises the band picker by selecting the `field` band of a `(time, band, y, x)` array), Autzen Stadium (a public COPC point cloud that keeps the point-cloud streaming path exercised on every fresh deploy), and two admin-boundary vector PMTiles "context layers" (countries and states/provinces, `connection_type="pmtiles"`, `tile_type="vector"`) that authors can drop on a story chapter as overlays. The two context-layer seeds point at live PMTiles built from geoBoundaries CGAZ (CC BY 4.0) and hosted in the `sandbox-data` R2 bucket under `context/` ā€” see [context-layers.md](context-layers.md) for the tippecanoe + R2 data-prep pipeline and the required attribution string. The COPC seed carries `connection_type="copc"`, WGS84 `bounds`, and a `config` of `{"color_mode": "elevation", "point_size": 2.0}`; it needs no remote probe. Unlike raster/vector datasets, COPC data is **stored to R2 but not registered in pgSTAC or tipg** ā€” the browser streams the `.copc.laz` directly via HTTP range requests (`maplibre-gl-lidar` + `copc.js`), so there is no tile server in the loop. Seeds that set `zarr_time_dim` trigger a startup probe that fetches and decimates the remote time coordinate, storing the result as `timesteps` in the connection config; if the probe fails, that seed is skipped. The probe currently only handles `int64` nanoseconds-since-epoch time coordinates, so seeds with other units (e.g. "days since 2024-01-01") must skip the probe by leaving `zarr_time_dim` unset and inline `timesteps` directly in `config` (as the Fields of The World seed does for its 2 timesteps). Seeds that set `geozarr_attrs` (the four single-resolution required GeoZarr keys) persist that override on the seeded row ā€” used for stores like IMERG that don't carry inline GeoZarr metadata. 4D variables can pre-set `extraDim` (the band/level/channel-style dim name) and `extraIndex` (the slice index) inside `config` to lock the rendered band. Add new `ExampleConnectionSeed` entries to grow the seed set.

When a curated field is added to an existing seed entry (for example, a `geozarr_attrs` override that didn't exist when the row was first seeded), the seeder backfills it onto the existing `is_example=True` row on the next startup. Backfill only writes fields that are currently NULL ā€” operator-set values (via `POST /api/connections/{id}/geozarr-attrs` or similar admin endpoints) are never overwritten. `config` and `name` are also preserved across backfills because they may carry runtime-probed timesteps or curator edits, and existing rows do not trigger a new remote time-probe. User-owned rows that share the seed's `(url, connection_type)` are left entirely untouched.
